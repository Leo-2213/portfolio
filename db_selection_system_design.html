// 00 — Before We Start

The Mindset Shift: Why Juniors Fail This

Let me be blunt with you. Most juniors fail the database selection question not because they don't know databases — they do. They fail because they treat it like a trivia quiz instead of an engineering discussion.

An interviewer at a company like Swiggy, Razorpay, Atlassian, or Flipkart isn't asking "which database is the best?" They're asking: "Can you reason through trade-offs like an engineer who ships real systems?"

When I ask a junior "which DB would you pick for this feature?", I'm not looking for the right answer. I'm looking at whether you ask the right questions back. A 2-year engineer who says "it depends — can you tell me more about the read/write ratio and consistency needs?" already beats 70% of candidates.

The Three Things Interviewers Actually Grade

  • Trade-off awareness — You know what you're giving up when you pick something
  • Scale intuition — Your choice makes sense at 1K, 1M, and 100M users
  • Justification depth — You can say why, not just what
// 01 — The Core Framework

How to Think About Database Selection

Before you name a single database, you need to ask yourself — or the interviewer — these questions. This is your decision tree. Never skip it.

The 7-Question Framework

  1. What is the data shape? — Is it tabular with fixed columns? Hierarchical JSON? Graph-like relationships? Time-stamped events? This alone eliminates 80% of wrong choices.
  2. What are the read vs write patterns? — Heavy reads (10:1)? Heavy writes (event streams)? Balanced? The answer changes everything. A write-heavy system and a read-heavy system solving the "same" problem need completely different databases.
  3. What is the consistency requirement? — Does every node need to see the same data immediately (strong consistency)? Or is eventual consistency acceptable? This is the CAP theorem entering the room.
  4. What are the latency expectations? — Sub-millisecond? Under 50ms? A few seconds is fine? If your interviewer says "user-facing real-time", think cache layers. If they say "batch reporting", think columnar.
  5. What scale are we designing for? — Start small, but explicitly say what the growth path looks like. 10K users today, 10M in 18 months changes which DB is viable.
  6. Are there complex queries or relationships? — Ad-hoc joins, aggregations, full-text search? Or are queries simple key-lookups? Complex queries love SQL. Simple lookups love NoSQL.
  7. What is the durability and compliance requirement? — Financial data? Medical records? These require ACID, write-ahead logs, and audit trails. A social media like/unlike does not.
Pro Tip

State these questions out loud in the interview. Don't just think them. It shows structured thinking and gives the interviewer a chance to steer you. Silence while you think looks like you're lost. Thinking aloud looks senior.

// 02 — Database Taxonomy

The Database Landscape (What Actually Matters)

Don't memorize a list. Understand the category first, then pick the specific tool. Here's the real breakdown:

1. Relational Databases (RDBMS)

When to use: Your data has clear relationships, you need ACID transactions, your schema is relatively stable, and you need complex queries. Think: anything where money or inventory changes hands.

PostgreSQL
Relational / RDBMS

The workhorse. Supports JSONB for semi-structured data, full-text search, GIS (PostGIS). Use this as your first pick unless you have a very specific reason not to.

MySQL / Aurora
Relational / RDBMS

Instagram ran on MySQL for years. Excellent read replicas. Aurora adds cloud-native scaling. Good default for web apps.

❌ Common Junior Mistake

Saying "NoSQL is better than SQL" or vice versa. They solve different problems. A 10x engineer at Stripe is still using PostgreSQL for payments in 2024. Don't fall for tech hype.

2. Document Databases (NoSQL)

When to use: Your data is naturally hierarchical (nested objects), schema changes frequently, you need horizontal scale from day one, and your queries are mostly by a single key or a few fields.

MongoDB
Document Store

Stores JSON-like documents. Great for product catalogs, user profiles, CMS data. Built-in sharding. Watch out for document size limits and lack of multi-document ACID (though it's improved).

DynamoDB
Document + Key-Value (AWS)

Serverless, near-infinite scale at the cost of extreme query rigidity. You must know your access patterns at design time. Used at Amazon for millions of req/sec. Hard to query flexibly later.

3. Key-Value Stores (Cache + Speed Layer)

When to use: Sub-millisecond access, session storage, rate limiting counters, leaderboards, pub/sub messaging. Always used as a layer in front of another database, not instead of it.

Redis
In-Memory Key-Value

Not just a cache. It has sorted sets (leaderboards), pub/sub, streams, distributed locks. Incredibly versatile. Data lives in RAM — fast but size-limited and expensive.

Memcached
In-Memory Cache

Simpler than Redis. Just a cache. Use when you need pure caching and multi-threaded is important. Redis has mostly replaced Memcached for new systems.

4. Wide-Column Stores

When to use: Massive write throughput at scale, time-ordered data, IoT sensor data, audit logs, activity feeds where you write a lot and read in a predictable pattern by key.

Apache Cassandra
Wide-Column Store

Netflix, Apple, Discord use this. Linear horizontal scale, no single point of failure, tunable consistency. Trade-off: NO joins, NO ACID transactions, query patterns must be defined upfront.

HBase
Wide-Column (Hadoop)

Lives in the Hadoop ecosystem. Good for batch + real-time on the same data. Less common in cloud-native stacks now.

5. Search Engines

When to use: Full-text search, fuzzy matching, autocomplete, faceted filtering. Do NOT use your main DB for LIKE '%query%' — it's a table scan at scale. Put this in a dedicated search layer.

Elasticsearch
Search Engine / Analytics

The standard. Inverted index for full-text search. Also used for log analytics (ELK stack). Synced from your primary DB via change data capture or events. Not your source of truth.

OpenSearch / Typesense
Search Engine

OpenSearch is AWS's Elasticsearch fork. Typesense is simpler and great for product search with typo tolerance.

6. Time Series Databases

When to use: Metrics, monitoring, IoT sensor readings, stock prices — any data where time is the primary dimension and you're doing range queries, aggregations, and downsampling.

InfluxDB
Time Series

Purpose-built for time-series. Handles high write throughput for metrics. Auto-retention policies to delete old data. Used in infrastructure monitoring.

TimescaleDB
Time Series on PostgreSQL

PostgreSQL extension. If your team knows SQL well, this is a pragmatic choice. Hypertables partition data by time automatically.

7. Graph Databases

When to use: Data where relationships ARE the data — social networks (friend-of-friend), recommendation engines, fraud detection (connected accounts), knowledge graphs.

Neo4j
Graph Database

Nodes and edges as first-class citizens. Cypher query language. Terrible for transactional workloads but brilliant for traversals. "Who are the friends of friends within 3 hops?" — SQL would be slow; Neo4j is native.

8. Data Warehouses (OLAP)

When to use: Analytics, reporting, BI dashboards. Optimized for reading large datasets, not for transactional writes. These are NOT your operational databases.

Snowflake / BigQuery
Cloud Data Warehouse

Columnar storage. A query scanning 1 billion rows takes seconds. Separate compute and storage. Cost can get wild if you're not careful.

Amazon Redshift
Cloud Data Warehouse

AWS ecosystem. Mature, well-integrated. Good for analytics teams on existing AWS infrastructure.

// 04 — Interview Q&A

Commonly Asked Questions (With Real Answers)

Q

Design a URL shortener like bit.ly. Which database would you choose and why?

// Strong Answer

First I'd ask: what's the read-to-write ratio? For a URL shortener, it's probably 100:1 or higher — millions of redirects for every URL created. That tells me reads are the bottleneck, not writes.

For the core data — shortcode → original URL mapping — this is a simple key-value lookup. The data is small (under 1KB per record), write-once read-many, and no complex queries are needed. I'd use a relational database like PostgreSQL as the source of truth because it's reliable, supports easy analytics (which URLs are most clicked), and the write volume is manageable.

But for the actual redirect path, I'd put Redis in front of it as a cache. Short codes that are popular get cached for 24 hours. 99% of traffic gets served from Redis at sub-millisecond speeds. PostgreSQL only gets hit for cache misses.

The schema is simple:

-- PostgreSQL: source of truth CREATE TABLE urls ( id BIGSERIAL PRIMARY KEY, short_code VARCHAR(8) UNIQUE NOT NULL, long_url TEXT NOT NULL, user_id BIGINT, created_at TIMESTAMPTZ DEFAULT NOW(), expires_at TIMESTAMPTZ ); -- Index on what we query by CREATE INDEX idx_short_code ON urls (short_code);
-- Redis: cache layer -- key: "url:{short_code}" → long_url -- TTL: 86400s (24 hours) for hot links SET url:abc123 "https://example.com/very-long-path" EX 86400

If the interviewer then says "now design the analytics — how many clicks per URL per day?" — that's when I'd add a Cassandra or a time-series DB for click events, or stream events through Kafka into a data warehouse for aggregation.

What made this answer strong

Started with a clarifying question (read/write ratio), justified each database choice separately, showed layered architecture, and proactively extended to the next likely follow-up.

Q

When would you NOT use a relational database? Walk me through a real situation.

// Strong Answer

Great question, and the honest answer is: fewer situations than the NoSQL hype would suggest. But there are real cases.

Case 1 — Schema is genuinely unpredictable. Say you're building a product catalog for an e-commerce site where different product categories have wildly different attributes. A TV has screen size, refresh rate, resolution. A t-shirt has size, color, fabric. You'd either need a giant table with hundreds of nullable columns, or an entity-attribute-value anti-pattern in SQL. A document store like MongoDB handles this naturally — each product document has only the fields relevant to its category.

Case 2 — Write throughput SQL can't keep up with. Imagine an activity feed system where 50 million users are liking, commenting, posting all day — that's potentially millions of writes per second across the platform. A single PostgreSQL primary hits its write ceiling. You'd need Cassandra or DynamoDB because they distribute writes across nodes without a bottleneck primary.

Case 3 — You need genuinely infinite horizontal scale. DynamoDB at Amazon handles 89 million requests per second at peak. You cannot shard a relational database to that scale without enormous complexity. At that point, you give up flexible querying and embrace the NoSQL constraint consciously.

❌ What Juniors Get Wrong

"NoSQL scales better, so just use that." — Wrong. NoSQL scales writes horizontally but sacrifices joins and transactions. If you need both scale AND transactions, look at Google Spanner or CockroachDB (distributed SQL). There's a middle ground.

Q

Explain the CAP theorem and how it affects your database choice in a real system.

// Strong Answer

CAP stands for Consistency, Availability, and Partition Tolerance. The theorem says: during a network partition (nodes can't talk to each other), you must choose between Consistency and Availability. You can't have both. Partition tolerance isn't optional in distributed systems — networks fail, so you always have P.

In practice this means:

  • CP systems (Consistency + Partition Tolerance) — Will refuse requests rather than serve stale data. Example: HBase, MongoDB (with strong consistency configured), Zookeeper. Use for: financial systems, inventory where overselling is catastrophic. You'd rather show an error than show wrong stock count.
  • AP systems (Availability + Partition Tolerance) — Will serve requests even if data might be slightly stale. Example: Cassandra, DynamoDB (eventually consistent). Use for: social media feeds, product recommendations, shopping carts. A slightly stale feed is fine. Going down is not.
When you're in an interview, don't just define CAP — apply it. Say something like: "For our payment processing service, I'd choose a CP system because a customer seeing their payment fail is better than charging them twice. But for the notification service showing unread count, AP is fine — if they see 5 instead of 6 for a second, no one cares."

Also mention PACELC if you want to stand out — it extends CAP to also consider latency vs consistency trade-offs even when there's no partition. Cassandra lets you tune consistency per query (ONE, QUORUM, ALL). That's real-world nuance interviewers love to hear.

Q

Design the database layer for a ride-sharing app like Uber or Ola. What databases would you use for different parts of the system?

// Strong Answer

A ride-sharing app has multiple distinct data domains that need different databases. Let me break it down by domain:

DomainData TypeDatabase ChoiceReason
User accounts, Driver profiles Structured, relational PostgreSQL ACID needed; users link to payments, rides, ratings
Payments, Earnings Financial transactions PostgreSQL + Stripe ACID non-negotiable; money must never be lost
Driver location tracking (live) Real-time geo updates (1/sec per driver) Redis + Geo In-memory, sub-ms lookup; Redis GEOADD/GEODIST handles proximity
Trip history, Ride events Append-only, time-ordered Cassandra High write volume; query by driver_id + time range = native partition key
Notifications, Real-time status Pub/sub, ephemeral messages Redis Pub/Sub or Kafka Driver accepted/ETA updates don't need persistence beyond delivery
Analytics — surge pricing model, heatmaps Aggregated event data BigQuery / Redshift OLAP queries on billions of trip events for BI team

The key insight here is that the driver location problem is a completely different database problem from the payment problem. Uber actually uses a custom geospatial system for matching, but Redis with its Geo commands is a reasonable interview-level answer for the location layer.

Follow-up: How do you keep these in sync?

Use event-driven architecture. When a ride completes, publish an event to Kafka. Consumers update the analytics DB, send notification, update billing. The databases are decoupled and updated asynchronously. This is the architecture pattern behind every large-scale app — not one DB, but a mesh of specialized stores synced via events.

Q

How would you design the database for a Twitter-like feed system?

// Strong Answer

This is a famous hard problem. There are two approaches — fan-out on write vs fan-out on read. Let me explain both and when to use which.

Approach 1 — Fan-out on write (Push model): When a user posts a tweet, immediately write it to the feed of every follower. Reading the feed is just a cache lookup.

-- When @virat posts a tweet: -- 1. Store tweet in primary DB (PostgreSQL) -- 2. Get all followers (PostgreSQL) -- 3. Write tweet_id to each follower's feed list LPUSH feed:user_id_123 tweet_id_999 -- Redis list LPUSH feed:user_id_456 tweet_id_999 -- ... for each follower (async, via Kafka)

Problem: Celebrities (Virat Kohli, Sachin Tendulkar) have 50 million followers. One tweet = 50M Redis writes. This is called the "celebrity problem" or "hot key problem".

Approach 2 — Fan-out on read (Pull model): Only store tweets in one place. When a user opens their feed, query the tweets of everyone they follow, merge them, sort by time. No pre-computation needed.

Problem: If you follow 500 people, reading the feed means 500 DB queries merged and sorted. Slow.

The real answer (hybrid — what Twitter actually does):

  • For regular users: fan-out on write. Pre-populate their feed in Redis.
  • For celebrities (>1M followers): fan-out on read. Don't pre-populate. When you request your feed, merge the pre-computed feeds of everyone you follow, and then fetch the last few tweets from any celebrities you follow separately and merge in real-time.
  • Storage layer: tweets stored in a distributed store (Cassandra), feed lists stored in Redis, user graph stored in PostgreSQL or a separate graph store.
This question is designed to see if you know that there's no perfect solution — only trade-offs. The interviewer wants to hear you say "it depends on whether the user is a celebrity or a normal user" and propose the hybrid model. That's the senior-level answer.
Q

How do you handle database scaling when your single PostgreSQL instance starts hitting limits?

// Strong Answer — Give this in layers

This is a step-by-step problem. You don't jump straight to sharding. Here's the exact progression I'd walk through:

  1. Add a read replica first. 80% of web app traffic is reads. A read replica immediately doubles your read capacity. Route all SELECT queries to the replica, all writes to primary. Almost no application changes needed. This solves most problems.
  2. Add a caching layer (Redis) in front of reads. For data that's read frequently but changes infrequently (user profile, product data), cache at the application layer. Cache hit ratio of 90% means your DB sees 10% of the traffic. Measure and cache your hottest queries.
  3. Vertical scaling. If you're on a 4-core 16GB machine, move to a 32-core 256GB machine. RDS/Cloud databases make this a config change. Often buys you 12-18 months. Cheap and underrated.
  4. Connection pooling (PgBouncer). PostgreSQL is heavy per connection. If you have 1000 app servers each opening 10 connections, that's 10,000 connections — PostgreSQL chokes. PgBouncer pools connections, multiplexing 10,000 app connections into 100 DB connections.
  5. Table partitioning. Before sharding across machines, partition within one machine. Orders table → partition by created_at month. PostgreSQL handles queries on the right partition automatically. Queries that filter by date are now 100x faster.
  6. Functional decomposition / sharding by service. Split the database by domain. Payments DB separate from Notifications DB separate from User DB. Different services own different schemas. This is "vertical sharding" — split by feature, not by row.
  7. Horizontal sharding (last resort). Split one table across multiple databases by a shard key (e.g., user_id % 4 → DB0, DB1, DB2, DB3). Complexity explodes: no cross-shard joins, distributed transactions are hard, rebalancing is painful. Only do this when everything else fails.
❌ Junior Mistake

Jumping straight to "I'd shard the database." Sharding is the nuclear option. Every step before it is cheaper, simpler, and should be exhausted first. An interviewer who hears "I'd add a read replica and Redis cache first" will be more impressed than someone who immediately says "horizontal sharding."

Q

Design a rate limiter. Which database/data store would you use?

// Strong Answer

A rate limiter needs to track how many requests a user/IP has made in a time window (e.g., 100 requests per minute). The data access pattern is: increment a counter on every request and check if it exceeds the limit — in a single atomic operation.

Redis is the obvious and correct choice here. Here's why:

  • Counters need atomic increment + check — Redis's INCR is atomic, no race conditions
  • TTL is native — set the key to expire after 60 seconds, no cleanup job needed
  • Sub-millisecond — adds no meaningful latency to the request path
  • Can be clustered for multi-region rate limiting
-- Fixed Window algorithm using Redis -- Key: "rl:{user_id}:{minute}" e.g., "rl:user_123:2024-01-15-10-32" FUNCTION is_allowed(user_id, limit=100): key = f"rl:{user_id}:{current_minute()}" count = INCR key -- atomic increment IF count == 1: EXPIRE key 60 -- set TTL on first request RETURN count <= limit -- allow or deny

If the interviewer asks about the Sliding Window algorithm (more accurate, avoids burst at window edges): use Redis Sorted Sets. Store each request's timestamp as a score, use ZRANGEBYSCORE to count requests in the last 60 seconds, remove expired entries with ZREMRANGEBYSCORE. Slightly more complex but much more accurate.

Why not a SQL database? Because rate limiting is on the hot path of every API request. Adding a SQL round-trip (even 5ms) to every request would add 5ms to every API call across your entire platform. At 100K req/sec, that's a bottleneck. Redis at 0.1ms is invisible.

Q

What is database denormalization and when would you use it intentionally?

// Strong Answer

Normalization removes data duplication — every piece of data lives in exactly one place. Denormalization intentionally duplicates data to make reads faster at the cost of write complexity.

When to denormalize:

  • Your most common query joins 5+ tables and runs thousands of times per second — denormalize by storing the computed result
  • You're using a NoSQL database that doesn't support joins (Cassandra, DynamoDB) — you have no choice but to denormalize
  • Analytics/reporting queries aggregate data that barely changes — pre-aggregate into a summary table

Real example — Swiggy/Zomato order history:

-- Normalized: to show order history, you need: -- orders JOIN order_items JOIN restaurants JOIN dishes -- That's 4 table lookups for every order history page load -- Denormalized order document (stored as JSON or in Cassandra): { "order_id": "ord_123", "restaurant_name": "Meghana Foods", // duplicated "restaurant_logo_url": "...", // duplicated "items": [ { "name": "Chicken Biryani", "price": 280 } // duplicated ], "total": 280, "ordered_at": "2024-01-15T19:30:00Z" } // Single document read = instant. No joins needed.

The trade-off: if Meghana Foods changes its logo, you'd normally update one row. With denormalization, you'd need to update every historical order document — but for an order history that never changes after completion, this is actually fine. The restaurant logo at the time of order is what matters historically.

Denormalization is one of the most misunderstood concepts. Juniors think "denormalization = bad practice." Seniors know it's a deliberate performance optimization. The key is knowing what invariants break and whether that's acceptable for your use case.
// 05 — Real Scenarios

Full System Scenarios — Walk Through My Thinking

🛒 Scenario: Design Flipkart's Flash Sale System
The product has 10,000 units. 1 million users hit "Buy Now" in the first 5 seconds.

This is one of the hardest database problems in e-commerce: inventory management under extreme concurrency. The danger: two users both see "1 unit left," both click buy, both get a confirmation, you've oversold. That's a real business disaster.

The Database Strategy

Step 1 — Redis for inventory counter (not PostgreSQL). Before the sale, move the inventory count to Redis. Why? PostgreSQL can handle maybe 10,000 writes/sec on a good machine. Redis handles 1,000,000 writes/sec. The atomic DECR command in Redis means no overselling:

-- Atomic decrement in Redis (no race conditions) FUNCTION reserve_item(product_id): key = f"inventory:{product_id}" remaining = DECR key -- atomic, thread-safe IF remaining < 0: INCR key -- rollback RETURN "sold_out" RETURN "reserved"

Step 2 — Queue the orders, don't process immediately. Don't write to PostgreSQL on the request path. When a user's Redis reservation succeeds, push to a Kafka queue. A consumer reads from Kafka and writes to PostgreSQL with proper ACID transaction (deduct inventory, create order, charge payment). This decouples the high-traffic request from slow DB writes.

Step 3 — What happens to PostgreSQL? It becomes the system of record, processing at its own pace. The eventual consistency is acceptable here — the user sees "Order Confirmed" immediately (based on Redis reservation) and gets a payment confirmation email a few seconds later when the queue processes it.

STEP 1
Redis DECR (atomic)
STEP 2
Kafka Queue push
STEP 3
Consumer → PostgreSQL
STEP 4
Payment + Confirmation
Key Insight

This is not "pick one database." It's using Redis for speed and atomic counting, Kafka for buffering, and PostgreSQL for durability. Knowing when each layer does its job is what separates a junior answer from a senior one.

📊 Scenario: Design Razorpay's Transaction History & Analytics
Show a merchant their last 10,000 transactions. Also power their analytics dashboard.

This scenario has two completely different workloads hiding in one question: OLTP (show me my transactions — fast, individual lookups) and OLAP (show me analytics charts — slow aggregations on millions of rows). The mistake is using one database for both.

OLTP Layer — Merchant Transaction Lookup

A merchant viewing their recent transactions is a simple query: SELECT * FROM transactions WHERE merchant_id = X ORDER BY created_at DESC LIMIT 100. This hits PostgreSQL with proper indexing. The index on (merchant_id, created_at DESC) makes this fast even with 100M rows.

But if a merchant has 5 million transactions and wants to search by customer name or payment method — that's full-text search. You'd sync to Elasticsearch and let them search there without hammering the transactional DB.

OLAP Layer — Analytics Dashboard

Queries like "What was my revenue by state in Q3?" or "Which payment method has the highest success rate?" require scanning millions of rows with aggregations. Running this on PostgreSQL would lock up your operational database.

Solution: Stream all transaction events to a data warehouse (BigQuery/Redshift) via a CDC pipeline (Debezium reads PostgreSQL WAL → Kafka → BigQuery). The analytics dashboard queries BigQuery, which is columnar and built for this exact workload — a query scanning 500M rows takes 3 seconds instead of 5 minutes.

-- PostgreSQL: OLTP query (fast, indexed) SELECT id, amount, status, created_at FROM transactions WHERE merchant_id = 98765 AND created_at > NOW() - INTERVAL '30 days' ORDER BY created_at DESC LIMIT 100; -- BigQuery: OLAP query (analytics, columnar scan — never run this on Postgres) SELECT DATE_TRUNC(created_at, MONTH) AS month, state, SUM(amount) AS total_revenue, COUNT(*) AS transaction_count FROM transactions_warehouse WHERE merchant_id = 98765 AND created_at BETWEEN '2024-07-01' AND '2024-09-30' GROUP BY 1, 2 ORDER BY 1;
🔍 Scenario: Design Swiggy's Restaurant Search & Discovery
Search "Biryani near me", filter by rating, diet, cuisine, price. Sort by relevance.

This is a search + geospatial problem. The temptation is to add LIKE queries to your main restaurant DB. Do not do this.

Why Not Just PostgreSQL?

A LIKE '%biryani%' query on 500,000 restaurants is a sequential scan. It doesn't use indexes and gets worse as you add filters (rating > 4, isVeg = true, distance < 5km). PostgreSQL's full-text search (tsvector) is better but still not built for fuzzy matching, typo tolerance, or relevance scoring.

The Right Architecture

Source of truth: PostgreSQL stores restaurant master data — menus, ratings, hours, address. Updated when restaurants onboard or update details.

Search layer: Elasticsearch gets a synced copy of restaurant data with GeoPoint fields. It handles:

  • Full-text search with fuzzy matching ("byriani" → "Biryani")
  • Geospatial distance filter — GEODISTANCE query filters restaurants within 5km of user
  • Multi-filter — cuisine, rating, dietary preference all applied as Elasticsearch filters
  • Relevance scoring — custom scoring formula: rating × recency × distance × sponsored boost
// Elasticsearch query concept { "query": { "bool": { "must": [ { "match": { "menu_items": { "query": "biryani", "fuzziness": "AUTO" } } } ], "filter": [ { "geo_distance": { "distance": "5km", "location": { "lat": 12.9716, "lon": 77.5946 } } }, { "range": { "rating": { "gte": 3.5 } } }, { "term": { "is_open": true } } ] } }, "sort": ["_score", { "delivery_time_mins": "asc" }] }

Sync strategy: When a restaurant updates their menu or gets new ratings, update PostgreSQL first (source of truth), then update Elasticsearch asynchronously via an event (Kafka consumer). Elasticsearch might be 1-2 seconds stale — that's fine for a search result.

// 06 — Failure Modes

Common Traps That Kill Interviews

!

Trap 1: Picking a database by familiarity, not fit

Saying "I'd use MongoDB because I know it well" is the fastest way to fail. The interviewer wants to see you pick based on requirements, not comfort. If you only know one database, learn at least the use cases for 5 categories so you can speak to trade-offs even if you haven't implemented them.

Never Say This

"I'd use MySQL because that's what I use at work." — Replace with: "Given the transactional requirements here, MySQL/PostgreSQL would be appropriate because..."

!

Trap 2: Ignoring the operational complexity of your choice

Every database has an operational cost. Cassandra is powerful but requires expertise to tune compaction, repair, and rebalance. If your team is 5 engineers, proposing Cassandra + Kafka + Elasticsearch for a startup's first version is over-engineering that will slow you down. Mention this trade-off: "At this scale, I'd start with PostgreSQL and Redis. Cassandra would make sense when we hit X write volume and have the ops team to support it."

Maturity Signal

Engineers who understand operational complexity — not just technical fit — show they've shipped and maintained real systems.

!

Trap 3: Not mentioning indexes when talking about query performance

A huge percentage of "my database is slow" problems are actually missing index problems. Whenever you design a table, explicitly talk about what indexes you'd create and why. This shows you think about query plans, not just schema.

-- Don't just say "I'd store orders in PostgreSQL" -- Say what indexes and why: CREATE INDEX idx_orders_user_created ON orders (user_id, created_at DESC); -- Covers: "show me user's recent orders" — composite index -- user_id filters first, created_at sorts in index (no sort step) CREATE INDEX idx_orders_status ON orders (status) WHERE status = 'pending'; -- partial index: only indexes active rows -- 95% of orders are 'completed' — partial index is tiny and fast
!

Trap 4: Treating the cache as the database

Redis is not your primary store (unless you're doing ephemeral data like session tokens). If your Redis goes down, you should be able to rebuild it from your primary database. Candidates sometimes design systems where losing the Redis instance means data loss. Always ask: "If this cache is lost, can I reconstruct it?" If the answer is no, Redis isn't a cache — it's a database being misused.

!

Trap 5: Not talking about data migration and schema evolution

Interviewers at product companies love this follow-up: "How do you add a new column to this table that has 500M rows without taking downtime?" The answer involves: adding the column with a DEFAULT value (instant in PostgreSQL 11+), backfilling in batches with a background job, then enabling the NOT NULL constraint after backfill. Not knowing this suggests you've never dealt with a large production database.

Phrases That Sound Senior

The same idea delivered differently signals different seniority levels. Here's a translation guide:

Junior PhrasingSenior Phrasing
"I'd use MongoDB because it's flexible." "Given that our schema is likely to evolve as we add product types, a document store gives us the flexibility to iterate without migrations. I'd use MongoDB here, though I'd define core fields as required."
"Redis is faster than a database." "For the session storage use case, the data is temporary, sub-millisecond access is required, and the data is reconstructible if lost — Redis is the right choice. I'd set TTLs appropriately and not treat it as a source of truth."
"We should use a NoSQL database to scale." "Before moving to NoSQL, I'd first evaluate whether read replicas, connection pooling, and a caching layer can meet our SLAs. NoSQL introduces operational complexity and constraints on queries. If we've exhausted SQL scaling options and the access patterns map well to NoSQL — specifically that they're predictable and don't need ad-hoc joins — then I'd explore Cassandra or DynamoDB."
"I'd add an index." "I'd add a composite index on (user_id, created_at DESC) to cover the most common access pattern. The order of columns matters — user_id has high cardinality and goes first, created_at sorts within that. I'd also monitor the index size since wide tables with many indexes can slow writes."
"We'd shard the database." "Sharding is complex and I'd resist it as long as possible. Let me first check whether the problem can be solved with read replicas, a CDN cache, query optimization, or functional decomposition. If we genuinely hit PostgreSQL's ceiling, I'd shard by user_id with consistent hashing to distribute evenly, but I'd flag that cross-shard queries become application-level joins."

The 60-Second Answer Template

Use this structure for any database selection question:

  1. Clarify requirements — "Let me ask a few questions: what's the read/write ratio, what's the scale expectation, and are there strict consistency requirements?"
  2. Identify data shape and access patterns — "The core data is [X] and the primary access patterns are [Y] and [Z]."
  3. State your choice with justification — "For the primary store, I'd use [DB] because [specific reason tied to their requirements]."
  4. Acknowledge the trade-off — "The trade-off with this choice is [X]. We accept this because [Y]."
  5. Add the supporting layers — "I'd complement this with a Redis cache for [hot reads/sessions/rate limiting] and Elasticsearch for [search functionality]."
  6. Address scale evolution — "At current scale this works. When we hit [threshold], we'd need to [next step — read replica, sharding, etc.]."
The interview is a conversation, not a presentation. Pause after your first DB choice and ask: "Does that direction make sense, or is there a constraint I should consider?" Showing you can collaborate under ambiguity is more impressive than having a perfect pre-planned answer. That's what real system design looks like.
